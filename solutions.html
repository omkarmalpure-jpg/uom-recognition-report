Proposed Solutions — Closing the Unrecognized Gap

Overview of the Unrecognized Population

The 53,754 rows that Step 5 could not resolve fall into three buckets, each requiring a different strategy:

Root cause Rows Strategy
No / zero purchase history 47 External enrichment (agentic search or catalogue match)
Very low purchase history (1–4 units) 31,400 Similar-product transfer + analyst review
Some history but price out of range 21,900 Threshold relaxation / cluster expansion

Solution A — Agentic Web Search for Zero-History Products

The problem

When a product has no purchase history at all, Step 5 has no anchors to compute proximity against — no DBSCAN cluster, no WAP, nothing. The only signal available is the product description itself.

Proposed approach

An agentic pipeline can augment these rows autonomously:

  1. Query construction — from RecognizedProductDesc, RecognizedManufacturerName, and RecognizedProductCatalogNumber build a structured search query (e.g., "Abbott 1192ANS Anchor SwiftLock unit of measure" or a medical-device pricing database lookup).

  2. Search & parse — the agent calls a web search tool or a medical product catalogue API and extracts candidate UOM labels and typical pack sizes.

  3. Citation attachment — every suggestion is stored with its source URL and snippet so an analyst can verify it before accepting.

  4. Confidence flagging — the agent rates its own confidence (e.g., “found exact catalog number” vs. “matched by description keyword”). Low-confidence suggestions are flagged for mandatory human review before write-back.

What needs to be built and tested

  • Prompt engineering to reliably extract structured UOM data from unstructured web pages.
  • A de-duplication pass to collapse near-identical suggestions.
  • A review UI that surfaces the citation alongside the suggested UOM for analyst sign-off.
  • A feedback loop: accepted suggestions are written back into the purchase history so future runs can resolve the same products without another search.

Trade-offs

Benefit Risk
Extends coverage to truly novel products Hallucination / incorrect web data — must require analyst approval
Provides auditable citations Adds latency and API cost per zero-history product
Self-improving over time as accepted results grow history Needs prompt fine-tuning per product category

Solution B — Similar-Product Transfer (Purchase History Bridge)

The problem

The 31,400 rows with very low purchase history have some data but not enough to build a reliable DBSCAN cluster or meaningful WAP.

Proposed approach

Rather than going to the web, look inside the existing purchase history for similar products:

  1. Similarity candidates — for a product with sparse history, find other products in the catalogue that share:

    • Same RecognizedManufacturerID
    • Same RecognizedProductTypeID
    • Similar description tokens (keyword overlap or embedding distance)

  2. UOM profile transfer — the similar product’s ThresholdV2 clusters and WAP are borrowed as a prior. The proximity scoring in Step 5 runs exactly as normal, just with the borrowed thresholds substituting for the absent ones.

  3. Downgraded confidence — because the thresholds are borrowed, the resulting confidence score is multiplied by a transfer discount factor (e.g., 0.80) to signal to downstream consumers that this is an inferred — not directly observed — assignment.

  4. Analyst review trigger — rows resolved via transfer are surfaced in a separate review queue so analysts can confirm the analogy is valid for that product category.

What needs to be discussed with analysts

  • Acceptable similarity criteria — does same manufacturer + same product type constitute a strong enough analogy? Are there categories where this would be systematically misleading (e.g., two products with the same type ID but very different pricing tiers)?
  • Discount factor calibration — how much should transferred confidence be penalised?
  • Segment sensitivity — certain clinical categories (e.g., implants vs. consumables) may require different similarity rules.

Trade-offs

Benefit Risk
Uses existing, trusted purchase data — no external calls Wrong analogy can propagate a systematic error
Fast — no added latency Similarity is hard to define without analyst input
Gracefully degrades to unrecognized if no similar product found Requires a new similarity index to be maintained

Solution C — Threshold Relaxation for Out-of-Range Prices

The problem

The 21,900 rows with adequate history fail because their file PriceUOM falls outside every DBSCAN cluster at a distance where the proximity score is below 0.30.

Common causes:

  • Supplier price change — the vendor raised or lowered prices since the last purchase, shifting the quote outside the historical band.
  • New pack size — the supplier is now selling in a quantity not previously ordered (e.g., BX/24 when history only has BX/12 and BX/100).
  • Currency / unit mismatch — a price-per-case was accidentally quoted as price-per-each.

Proposed approaches

C1 — Widened acceptance window (lower LOG_EPS threshold)

LOG_EPS = 0.25 currently corresponds to a roughly 1.8× tolerance from the cluster edge. Increasing it to 0.40 extends tolerance to ~2.5×. This is a global change and must be evaluated on a hold-out set to measure false-positive impact.

C2 — Price-trend adjustment

For products with a time-stamped purchase history, the system can detect upward or downward price trends and project the cluster range forward. A product whose EA price has trended from $10 → $13 → $16 over 12 months is likely priced around $19 today; the cluster built on 12-month-old data will be offset.

C3 — Analyst escalation queue

Rows where the best candidate score is 0.20–0.30 (just below acceptance) are surfaced in a dedicated review queue. The analyst sees the best DBSCAN candidate, its score, and the distance from the cluster edge — making a quick accept/reject decision fast. Accepted overrides are recorded and used to retrain the threshold or expand the cluster.

Prioritisation Matrix

Solution Potential reach Implementation effort Requires analyst input
A — Agentic search 47 rows (zero-history) High — prompt engineering + review UI Yes — citation sign-off
B — Similar-product transfer 31,400 rows (low history) Medium — similarity index + discount logic Yes — similarity criteria definition
C1 — Widen LOG_EPS Subset of 21,900 rows Low — one config change + validation No — but needs hold-out evaluation
C3 — Escalation queue Near-miss rows (score 0.20–0.30) Low — UI change only Yes — per-row review

Recommended Starting Point

  1. Immediate — implement C3 (escalation queue) to capture near-misses with minimal engineering. This recovers the easiest unrecognized rows and builds analyst feedback data that can be used to calibrate all other solutions.

  2. Short term — evaluate C1 (LOG_EPS relaxation) on a hold-out set. If the false-positive rate is acceptable, deploy it to recover a large portion of the out-of-range rows automatically.

  3. Medium term — design and pilot Solution B (similar-product transfer) with analysts. The similarity criteria need to be co-designed with domain experts before any automated write-back.

  4. Longer term — prototype Solution A (agentic search) on a small batch of zero-history products, evaluate citation quality, and build the review workflow before scaling.
